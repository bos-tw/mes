# 二次篩選實作計畫（2026-06-25）

## 文件定位

本文件是二次篩選的主要實作計畫。後續若 `shipping-phase1-todo-2026-06-22.md`、`shipping-phase2-implementation-plan-2026-06-22.md` 或其他舊文件對二次篩選有不同描述，以本文件與 `second-screening-todo-2026-06-25.md` 為準。

相關待辦清單：`docs/second-screening-todo-2026-06-25.md`

## 核心定義

二次篩選是一個可獨立追溯的品質流程節點，不等同於退貨後重篩，也不應只靠生產工單備註記錄。

系統需要支援同一個二次篩選模組下的兩種理由：

- `relaxed_after_high_defect`：第一次篩選後不良過多，回報客戶，客戶同意放寬標準後再做第二次篩選，合格品放行送回或出貨。
- `customer_required_second_pass`：客戶或訂單規格要求每批固定二次篩選，這是正常製程要求，不是異常或退貨流程。

兩者都是二次篩選，差異在於來源理由與必要佐證，不應拆成兩套不相干的功能。

## 設計原則

- 一批貨最多只記錄一次二次篩選，不設計第三次、第四次遞迴篩選。
- 生產工單頁必須能直接看出此批貨是否有二次篩選。
- 生產工單頁必須能直接看出二次篩選原因、案件編號與結果摘要。
- 二次篩選必須有自己的篩選紀錄，不可只用工單備註、狀態或單一數字帶過。
- 二次篩選必須追溯到原始訂單、原始工單、第一次篩選結果、第二次篩選標準與第二次篩選結果。
- 退貨單只能是來源之一，不可作為二次篩選唯一入口。
- `rescreen_type` 只描述標準型態，例如嚴格或放寬；業務理由必須由 `second_screening_reason` 之類欄位表示。

## 實作規範

- 不採「最小可行版本」作為交付標準。
- 每次實作都必須朝完整二次篩選功能閉環推進。
- 可以分批施工，但每一批都必須補齊該範圍內必要的資料模型、API、畫面、追溯、驗證與文件更新。
- 不交付只有欄位、只有畫面、只有 API 或只有備註的半套功能。
- 若某一批工作因風險需要拆小，必須清楚標示它屬於哪個完整功能閉環，並同步更新 todo 的剩餘缺口。
- 實作完成定義以現場流程能被使用者理解與追蹤為準，不以「程式可執行但流程未完整」為準。

## 現況問題

目前第一輪落地內容仍偏向「退貨後二次重篩」：

- `rescreen_batches` 建立流程主要從退貨單進入。
- UI 文案仍使用「二次重篩歷史紀錄」。
- `rescreen_type = strict_rescreen / relaxed_rescreen` 被拿來承擔過多語意，缺少真正的二次篩選理由。
- 生產工單雖有 `source_rescreen_batch_id`，但目前主要是執行工單回掛案件，還不是原始工單上的一眼可見狀態。
- 從生產工單建立二次篩選案件的正式 UI 尚未完成。

## 目標模型

既有 `rescreen_batches` 可以保留，但語意調整為「二次篩選案件」。

建議核心欄位語意：

- `rescreen_batch_number`：二次篩選案件編號。
- `second_screening_reason`：二次篩選理由。
- `rescreen_type`：標準型態，例如嚴格、放寬。
- `source_work_order_id`：原始生產工單。
- `rescreen_work_order_id`：第二次篩選執行工單。
- `source_order_id` / `source_order_item_id`：來源訂單與品項。
- `source_return_order_id` / `source_return_order_item_id`：退貨來源，僅在退貨來源存在時使用。
- `source_defect_history_record_id`：第一次篩選後造成二次篩選決策的不良紀錄，主要用於 `relaxed_after_high_defect`。
- `customer_approval_reference`：客戶同意放寬標準的備註、核准資訊或附件參考。
- `source_requirement_id`：客戶或訂單固定二次篩選要求來源，主要用於 `customer_required_second_pass`。

## 篩選紀錄

二次篩選詳情必須呈現完整篩選紀錄：

- 第二次篩選套用的服務項目。
- 原標準快照。
- 第二次篩選標準快照。
- 第二次篩選各服務的不良數量。
- 第二次篩選良品數量。
- 第二次篩選再次不良數量。
- 報廢或其他處置數量。
- 測量/篩選人員。
- 測量/篩選時間。
- 備註與附件。

固定二次篩選不一定有放寬後標準，但仍要保存第二次篩選使用的標準快照。

放寬標準重篩必須清楚保存原標準與放寬後標準差異。

## 工單端顯示

生產工單列表、詳情或編輯畫面需要顯示：

- 是否有二次篩選。
- 二次篩選原因。
- 二次篩選案件編號。
- 第二次篩選執行工單。
- 結果摘要：良品數、再次不良數、報廢/其他處置數。
- 二次篩選詳情入口。

這是本輪設計最重要的使用者體感：現場從生產工單就要能看懂這批貨有沒有經過二次篩選。

## 建立入口

二次篩選案件需支援三種建立入口：

- 從生產工單建立：用於客戶要求固定二次篩選。
- 從第一次篩選結果或不良紀錄建立：用於不良過多、客戶同意放寬標準。
- 從退貨單建立：用於退回品需要二次篩選時，作為可追溯來源之一。

## 實作順序

1. 文件與用詞收斂
   - 將主要追蹤文件收斂成本 plan 與 todo。
   - UI 文案從「二次重篩」調整為「二次篩選」。

2. 資料模型修正
   - 新增 `second_screening_reason`。
   - 補必要欄位或對現有欄位做語意調整。
   - 若新增 migration，同步更新 `tools/sync-local-schema.ps1` 的 `$migrationChecks`。

3. API 與建立流程
   - 允許從生產工單建立二次篩選案件。
   - 保留退貨單建立流程，但調整為來源之一。
   - 建立時依理由驗證必要欄位。

4. 工單端整合
   - 工單列表顯示二次篩選狀態與原因。
   - 工單詳情/編輯畫面顯示案件編號、結果摘要與導頁。
   - 原始工單與二次篩選執行工單都要能互相追溯。

5. 二次篩選紀錄完整化
   - 詳情頁呈現第二次篩選服務明細、標準快照與結果。
   - 再次不良正式進入不良品歷史。
   - 新庫存來源鏈寫入並可查。

6. 驗證與治理
   - 修改功能模組前後跑 `node tools/audit-system-health.js --changed --base origin/main`。
   - 涉及 DataSync 時跑 `node --check js/data-sync.js`、`node --check tools/audit-data-sync.js`、`node tools/audit-data-sync.js --write docs/data-sync-audit.md`。
   - 若修改配置型模組，跑 `node tools/validate-config-modules.js`。

## 2026-06-25 本輪落地範圍

本輪先完成「生產工單可發起固定二次篩選，退貨單仍可發起放寬後二次篩選，且工單列表可追溯」這一段閉環。

- 資料模型：新增 `second_screening_reason`、`customer_approval_reference`、`source_requirement_id`、`source_defect_history_record_id`，並將退貨來源欄位改為可空，讓非退貨來源的二次篩選可被正式記錄。
- API：`rescreen_batches/index.php` 可依來源自動分流到退貨單或生產工單建立流程；工單來源強制使用 `customer_required_second_pass`，退貨來源預設使用 `relaxed_after_high_defect`。
- 追溯：工單列表與單筆 API 回傳二次篩選案件、原因、狀態與執行工單；二次篩選詳情時間線可區分退貨來源與工單來源。
- UI：二次篩選模組用語從「二次重篩」收斂為「二次篩選」，新增原因與客戶佐證欄位；工單列表新增二次篩選欄，可建立或檢視案件。

- 二次篩選詳情補測量/篩選人員與時間的正式呈現。
- 側邊欄/權限顯示名稱從「二次重篩歷史紀錄」改為「二次篩選紀錄」。
