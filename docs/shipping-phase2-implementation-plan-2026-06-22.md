## 1. 第二階段目標

第一階段已完成可運作的出貨單骨幹，但仍以「摘要可用、流程可跑」為主。

第二階段目標不是重做第一階段，而是在不破壞現場已可使用流程的前提下，補齊下列能力：

- 出貨品質檢驗可被查詢、建立並與出貨單關聯。
- 不良品回送可追溯原工單、原庫存、原出貨與後續退貨重篩。
- 客戶載具不再只是一張出貨單上的快照，而是能顯示來源紀錄並分析是否仍有遺留載具在公司。
- 出貨、退貨、部分入庫、工單、檢驗之間可構成完整追溯鏈。

第二階段建議改採「先追溯、後守門」節奏：

1. 先把追溯入口與總覽補齊，讓現場能先看得到完整來源與流向。
2. 再定義不良品與二次重篩的資料口徑，避免後續返工。
3. 再建立客戶載具紀錄與遺留分析，不一開始擴大到總帳與財務層。
4. 二次重篩不走簡化版，直接以完整案件 / 規則 / 不良 / 新庫存來源鏈模型設計。
5. 最後才評估是否把品質檢驗與流程守門升級為硬阻擋。

## 2. 第二階段完成定義

本階段最低可交付結果：

- 使用者可從出貨單直接看到或開啟對應的出貨品質檢驗。
- 使用者可從側邊欄「二次重篩歷史紀錄」直接搜尋、檢視退貨後重篩歷程，不需理解資料表或工程欄位。
- 不良品回送資料可以回推：
  - 原工單
  - 原庫存
  - 原出貨單
  - 後續退貨 / 二次重篩
- 客戶載具可顯示進貨、工單、出貨相關紀錄，並可分析是否仍有遺留載具在公司。
- 出貨詳情頁可顯示主要來源與後續流向。
- 第二階段新增能力不強迫改寫第一階段既有資料輸入方式，需與既有測試流程相容。
- 品質檢驗、追溯鏈與總帳第一輪先以「可視、可查、可跳轉」為主，不先導入高摩擦硬守門。
- 二次重篩第一版不採資料簡化策略，需直接保留完整來源鏈，避免後續追溯斷層。
- 遠端使用者即使開啟 `index.html`，也必須被導向 `index.php` 主入口；前端版本偵測不得因靜態入口固定版本值而反覆顯示更新提示。

## 4. 主題一：跨模組追溯鏈總覽

### 4.1 背景

第二階段會同時碰到品質檢驗、不良回送、退貨、庫存、部分入庫與載具總帳。若沒有一個共用追溯總覽，後續每塊功能都會各自做入口，最後 detail 頁會變得零散。

### 4.2 第二階段目標

- 出貨詳情頁集中顯示主要來源與後續流向。
- 先以可查詢、可跳轉為主，不在這一階段就加入強硬守門。
- 後續品質檢驗、不良歷史、退貨、載具總帳都能掛在同一個追溯區塊下。

目前進度補充：

- 已在出貨單詳情集中顯示不良品歷史、品質檢驗、退貨單、來源工單、來源庫存等入口。
- 已補上退貨單導頁 context，可直接定位原出貨單相關退貨資料。
- 已把客戶載具紀錄 / 遺留分析與二次重篩案件入口併入同一個追溯區塊脈絡。

### 4.3 建議實作

- 在出貨單詳情建立獨立追溯區塊，至少整合：
  - 訂單
  - 工單
  - 部分入庫
  - 庫存項目
  - 出貨品質檢驗
  - 退貨單
  - 不良品歷史
- 入口先以摘要數字 + 跳轉按鈕為主，不要求第一輪就全部展開完整明細。
- 保留客戶載具總帳與二次重篩工單的未來掛點。

## 5. 主題二：出貨品質檢驗整合

### 5.1 背景

目前實務上尚未全面執行出貨品質檢驗，但系統上已需要預留，以便未來從「可選」升級為「應做」甚至「必做」。

### 5.2 第二階段目標

- 出貨單可顯示是否已有品質檢驗。
- 使用者可從出貨單直接建立或開啟品質檢驗。
- 檢驗結果可作為後續守門依據，但第二階段初期先不強制卡流程。

目前進度補充：

- 已完成出貨單詳情 / 建立編輯 modal 與出貨品質檢驗之間的顯示與導頁整合。

### 5.3 建議實作

- 在出貨單詳情區加入：
  - 檢驗筆數
  - 最近檢驗日期
  - 檢驗狀態
- 在出貨單操作列或詳情中加入「品質檢驗」入口。
- 第一輪先維持單頭關聯，逐項綁定出貨品項列為設計確認項，不急著在本輪擴大。
- 若最新檢驗為 `fail` 或尚未檢驗，可先做醒目提示，不先硬阻擋。

## 6. 主題三：不良品追溯與回送口徑

### 6.1 背景

不良品會隨出貨單回送客戶，之後可能再退回並進行二次重篩，因此不能只停在第一階段的摘要記錄。這一段最大的風險不是 UI，而是資料口徑未先定義。

### 6.2 第二階段目標

- 每筆出貨不良品摘要能回推來源工單與來源庫存。
- 若後續發生退貨與重篩，可追到第一次出貨單。
- 不良品歷史查詢可跨工單、出貨、退貨、重篩瀏覽。
- 不良品歷史紀錄可看出是否有標註在出貨單，以及是否需隨貨送回、是否已隨貨送回。

目前進度補充：

- 已在不良品歷史紀錄顯示出貨單標註、需隨貨送回、已隨貨送回、對應出貨狀態。
- 已補上不良品歷史到退貨單 / 出貨單的導頁入口。

### 6.3 建議實作

- 先定義不良品摘要來源欄位的必填 / 選填規則。
- 明確指定主追溯欄位與優先順序：
  - `source_work_order_id`
  - `source_inventory_item_id`
  - `source_shipping_order_id`
- 在 `不良品歷史紀錄` 中補足與出貨單相關的顯示欄位：
  - 是否有標註在出貨單
  - 是否需隨貨送回
  - 是否已隨貨送回
- 上述三個狀態需先定義判定口徑：
  - 何種情況視為「有標註在出貨單」
  - 何種情況視為「需隨貨送回」
  - 何種情況視為「已隨貨送回」
- 二次重篩若再產生不良，不可只混入一般工單不良備註，需視為正式不良來源之一：
  - 建議新增 `rescreen_batch_defects`
  - 或至少建立等價的正式來源表，再由 `不良品歷史紀錄` 統一彙整
- `不良品歷史紀錄` 後續需能分辨：
  - 首次生產不良
  - 出貨回送不良
  - 二次重篩再次發生的不良
- 二次重篩不良應保留上游追溯欄位，例如：
  - `source_rescreen_batch_id`
  - `source_return_order_item_id`
  - `source_defect_history_record_id`
  - `source_work_order_id`
  - `rescreen_round`
- 每次重篩產生的新不良都應新增新的歷史事件，不覆蓋原始不良紀錄。
- 規劃退貨後二次重篩的來源控制欄位，例如：
  - `source_shipping_order_id`
  - `source_return_order_id`
  - `rescreen_work_order_id`
- 等口徑確認後，再擴充 UI 與守門規則。

## 7. 主題四：客戶載具紀錄與遺留分析

### 7.1 背景

第一階段只記錄「本次出貨隨貨歸還了哪些載具」。目前你的需求是先做到「紀錄可看、來源可追、能判斷是否仍有遺留載具在公司」，不先做完整總帳。

### 7.2 第二階段目標

- 可從訂單撈取進貨時的載具紀錄。
- 工單與出貨單的載具記錄可被顯示、比對與追溯，但不強制要求一致。
- 可查詢並顯示某客戶的載具相關紀錄。
- 系統可自動分析目前該客戶是否仍有遺留載具在公司。

目前進度補充：

- 已先在出貨單詳情顯示客戶載具紀錄與遺留分析摘要。
- 現階段分析口徑為：訂單載具設定視為進場紀錄，出貨單載具摘要視為歸還紀錄。
- 已補上工單編輯視窗與客戶編輯視窗的客戶載具紀錄 / 遺留分析顯示。
- 工單與客戶畫面目前仍屬第一輪摘要顯示，尚未拆成獨立的客戶載具總覽頁。
- 已在出貨單詳情補上訂單載具設定 / 工單載具 / 部分入庫載具 / 出貨摘要的對照表，供第一輪追溯比對。
- 目前已能顯示「可能仍遺留在公司」的分析摘要，但尚未形成正式客戶載具流水帳。

### 7.3 建議實作

- 以「紀錄」為主，不先建立完整財務或結餘總帳模型。
- 先整理可用來源：
  - 訂單進貨時的載具紀錄
  - 工單中的載具記錄
  - 出貨單上的載具摘要
- 工單中的載具記錄與出貨單上的載具摘要只需可比對、可追溯，不設強制一致規則。
- 原因：現場可能同時有多批進來、載具不同，但同批出貨時可能合併；也可能同批進來後分批或不同載具出去。
- 針對客戶維度輸出載具紀錄摘要，並推算：
  - 已進場
  - 已隨貨歸還
  - 目前可能仍遺留在公司
- 第一輪明確不擴大到財務應收應付與正式總帳結算。

## 8. 主題五：退貨後二次重篩管制

### 8.1 背景

退貨產品會進行二次重篩，因此退貨單不應只是單獨事件，而應能回接第一次出貨及後續新工單。但這一段依賴前面不良品來源口徑先定義完成。

這裡不建議只靠一般工單去追來源，否則之後會變成一張工單一張工單地翻，追溯成本會越來越高。第二階段應把「二次重篩案件」獨立成正式資料模型，再由工單負責執行。

### 8.2 第二階段目標

- 退貨單可明確指出原出貨單來源。
- 若進入二次重篩，可識別是「重篩流程」而非一般新訂單。
- 二次重篩結果可回推第一次出貨與退貨。
- 二次重篩不可只靠一般工單備註或單一來源欄位，需能從案件主表直接看到來源、明細、執行工單與最終流向。
- 二次重篩完成後形成的新庫存，必須保留完整來源鏈，不可直接回寫原庫存，也不可只留下單一來源欄位。

### 8.3 設計口徑修正：二次篩選不是只等於退貨後重篩

2026-06-25 重新確認：本節原先把二次重篩過度綁定在退貨單與不良回流，設計口徑需要修正。

「二次篩選」應是同一個可追溯品質流程節點，支援兩種業務理由：

- `relaxed_after_high_defect`：第一次篩選後不良過多，回報客戶，客戶同意放寬標準後再篩一次，合格品放行送回或出貨。
- `customer_required_second_pass`：客戶標準嚴格，每批訂單都要求固定二次篩選，這屬於正常訂單/製程要求，不是退貨流程。

設計原則：

- 一批貨只需要記錄是否做過二次篩選，不設計任意多輪重篩。
- 二次篩選可維持獨立模組，但必須能從生產工單直接看到是否二次篩選、二次篩選理由、案件編號與結果摘要。
- 二次篩選必須有篩選紀錄，包含第二次篩選標準快照、篩選服務明細、良品數、再次不良數、處置、測量/篩選人員與時間。
- 退貨單只能是二次篩選的可能來源之一，不可再作為唯一入口。
- `rescreen_type = strict_rescreen / relaxed_rescreen` 只能描述標準型態，不能取代業務理由；需新增或改名為 `second_screening_reason` 一類欄位。
- 詳細修正設計見 `docs/second-screening-design-review-2026-06-25.md`。

### 8.4 建議實作

- 第二階段改採「二次篩選案件主檔 + 明細 + 執行工單/階段」模型：
  - `rescreen_batches`
  - `rescreen_batch_items`
  - `rescreen_batch_rules`
  - `rescreen_batch_defects`
  - `inventory_item_sources`（或等價正式來源鏈模型）
  - `work_orders.source_rescreen_batch_id`
- `rescreen_batches` 既有資料表可保留，但語意需調整為二次篩選案件追溯主體，至少承載：
  - `rescreen_batch_number`
  - `rescreen_type`
  - `second_screening_reason`
  - `customer_id`
  - `source_return_order_id`
  - `source_return_order_item_id`
  - `source_order_id`
  - `source_order_item_id`
  - `source_work_order_id`
  - `source_shipping_order_id`
  - `status`
  - `received_at`
  - `reason`
  - `decision_notes`
- 入口需支援：
  - 從生產工單建立：客戶要求固定二次篩選。
  - 從第一次篩選結果/不良紀錄建立：不良過多，客戶同意放寬標準。
  - 從退貨單建立：退回品需要二次篩選時，作為來源之一。
- 舊第一版假設需修正：
  - 不可再假設二次篩選與退貨單固定 1 對 1。
  - 不可再假設必須從退貨單建立。
  - 主檔直接保存原始訂單、原始工單
  - `rescreen_type` 可保留 `strict_rescreen` / `relaxed_rescreen`，但需與 `second_screening_reason` 分開
  - 使用獨立案件編號
- `rescreen_batch_items` 作為來源明細，至少可掛：
  - 退貨單品項
  - 不良品歷史紀錄
  - 出貨不良摘要
  - 必要時的來源庫存項目
- `rescreen_batch_rules` 作為二次重篩規則快照，特別用於 `relaxed_rescreen`：
  - `source_screening_service_id`
  - `original_tolerance_plus`
  - `original_tolerance_minus`
  - `original_ppm_standard`
  - `adjusted_tolerance_plus`
  - `adjusted_tolerance_minus`
  - `adjusted_ppm_standard`
  - `adjustment_reason`
- `strict_rescreen` / `relaxed_rescreen` 共用同一套主模型，但需能在畫面與歷史追溯中清楚區分標準型態；真正的二次篩選理由由 `second_screening_reason` 類欄位表示。
- `rescreen_batch_defects` 作為二次重篩再次發生的不良來源，至少承載：
  - `rescreen_batch_id`
  - `work_order_id`
  - `source_return_order_item_id`
  - `source_defect_history_record_id`
  - `defect_quantity`
  - `weight_per_unit_g`
  - `total_weight_kg`
  - `disposition`
  - `recorded_at`
  - `notes`
- `work_orders` 只作為執行載體：
  - 新增 `work_order_type = rescreen`
  - 新增 `source_rescreen_batch_id`
- 二次重篩完成後形成的新庫存，不採簡化欄位拼湊，應建立正式來源鏈：
  - 新庫存本身保留 `receipt_type = rescreen`
  - 來源鏈至少需能回到：
    - `source_rescreen_batch_id`
    - `source_rescreen_batch_item_id`
    - `source_return_order_id`
    - `source_return_order_item_id`
    - `source_shipping_order_id`
    - `source_work_order_id`

## 10. 風險與注意事項

- 不可破壞第一階段已可運作的出貨流程。
- 第二階段若直接引入強制守門，容易卡住現場測試，因此應先顯示、後阻擋。
- 客戶載具若一開始就做成完整總帳，範圍會快速擴大；目前應維持在紀錄、比對與遺留分析層級。
- 不良品回送與二次重篩若未定義來源欄位，後續會很難追。
- 若二次重篩只靠工單欄位或備註硬記來源，後續查詢會變成跨多張工單人工拼接，不適合作為正式追溯依據。
- 若二次重篩新庫存沒有正式來源鏈，而只保留單一欄位或回寫原庫存，會直接造成追溯斷層。
- 若「需隨貨送回」與「已隨貨送回」未先定義判定規則，歷史紀錄會出現語意不一致。
- 若未先整理追溯總覽，之後每個模組各做入口，會讓詳情頁結構持續碎裂。
- 主入口頁若有任何側邊欄、模組載入或導頁調整，`index.html` 與 `index.php` 必須維持同步，避免本機 / 遠端入口出現不同版本。
- 側邊欄顯示差異不一定是 HTML 檔案未同步，也可能是前端權限過濾與後端權限守門不同步；已將 `script.js` / `api/bootstrap.php` 權限映射一致性納入系統健康審計。
- 後續新增側邊欄模組時，必須同時通過雙入口同步檢查與前後端權限映射同步檢查，不可只憑畫面人工確認。
- 這次防再犯規則已同步寫入 `.github/copilot-instructions.md`；後續每輪初始化讀取規範時必須依此排查。
- 封閉測試更新包 `v3.0.7` 已以 safe build 產生，打包工具已驗證所有主檔與 migration 皆存在於 ZIP 中。
