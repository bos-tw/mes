# 手機版生產工單功能規格草案

更新日期：2026-06-20
狀態：第一版規格已定案；本機功能已大致落地，待遠端驗證與實機回歸
範圍：手機 / 平板友善的生產工單執行端

## 1. 目標與定位

- 手機版僅針對 `生產工單` 場景，不包含其他模組。
- 手機版不是桌面版功能縮小，而是現場人員的工單執行介面。
- 設計重點是讓現場人員能在機台旁快速查單、開工、部分完工、完工、拍照與回報異常。
- 介面必須手機友善，但視覺風格、登入樣式、文案語氣需與桌面版一致。
- 手機版入口獨立，但必須清楚標示為「手機版」。

## 2. 本輪已確認產品決策

- 手機版只做生產工單，不含其他模組。
- 完工照片為建議上傳，不強制；因部分工單不需要照片。
- `部分完工` 為第一版必做。
- 現階段權限先不做角色差異化，相關人員可操作內容先視為一致。
- 手機入口採獨立模組，但登入畫面樣式需與桌面版一致。
- 整體 UI 需以手機 / 平板友善方式開發，風格需與桌面版一致。
- 遠端主機正在進行實際上線測試中，本輪資料結構調整以 `只新增、不取代、不中斷舊流程` 為最高原則。
- `work_order_images` 既有用途先保留，不在本輪重新定義為通用附件中心。
- 本輪先不導入 `工單附件分類與歸屬模型` 的通用架構，避免影響既有桌面版與測試流程。
- 工單現場新照片需求先以獨立資料表處理：
  - `work_order_completion_images`
  - `work_order_defect_images`
  - `work_order_tool_condition_images`
- 手機版上傳新照片後，桌面版工單明細必須可同步讀取；桌面版第一階段先做唯讀整合，不急著提供編輯或搬移能力。

## 3. 使用情境

### 3.1 現場共用工單池

- 現場人員會互相支援，不採 `我的工單` 模式。
- 所有人員從共用工單清單進入，再依工單狀態、機台、日期等條件篩選。
- 工單可由不同人員接手回報，但每次操作必須保留操作者與時間紀錄。

### 3.2 QR Code / 快速入口

- 工單可提供 QR Code 或直接連結。
- 現場人員掃碼後可直接開啟對應工單的手機執行頁。
- QR Code 入口是加速方式，不取代手機版的工單清單入口。
- 第一版納入 QR Code / 直達工單入口規劃。

## 4. 功能範圍

### 4.1 第一版包含

- 手機版獨立入口
- 與桌面版風格一致的登入頁
- 生產工單清單
- 生產工單執行頁
- 開工回報
- 部分完工回報
- 完工回報
- 異常回報
- 完工照片上傳
- 不良品照片上傳
- 載具狀況圖片上傳
- 基本操作紀錄追溯

### 4.2 第一版不包含

- 訂單、出貨、庫存、退貨等其他模組的手機版
- 角色差異化權限流程
- 手機端建立工單主檔
- 排程編輯與複雜派工
- 離線模式
- 主管審核 / 班長退回修正流程

## 5. 入口規劃

### 5.1 模組命名建議

- 模組代號建議：`mobile_work_orders`
- 顯示名稱建議：`生產工單手機版`

### 5.1.1 專案目錄與正式路徑

- 手機版相關開發內容集中放在專案根目錄 `mobile/`。
- 未來正式使用路徑為：`https://mes.sort.com.tw/mobile`
- 因此手機版頁面、資源引用、API 呼叫與相對路徑規劃，需從一開始就以 `/mobile` 為部署前提。

### 5.2 入口形式

- 桌面首頁可新增 `生產工單手機版` 入口。
- 入口與既有桌面工單模組分開，避免桌機 / 手機邏輯混雜。
- 若後續需提供單獨網址，可規劃獨立進入路徑，但登入樣式沿用桌面版視覺。

### 5.3 入口顯示原則

- 必須清楚標示為手機版，避免與桌面版混淆。
- 入口文案需讓使用者一眼知道此入口是給手機 / 平板現場操作使用。

## 6. UI / UX 原則

- 採手機優先設計，平板次之。
- 維持與桌面版一致的品牌色、字體語氣、元件風格與登入樣式。
- 版面必須適合單手操作，主要按鈕需明顯且易點擊。
- 盡量減少欄位數與步驟數，不把桌面版大量資訊直接搬上手機。
- 同頁優先顯示本次操作必需資訊，其餘資訊可折疊或次要呈現。
- 回報成功 / 失敗的提示要清楚，避免現場重複送出。

## 7. 頁面規劃

### 7.1 登入頁

目的：
- 提供手機 / 平板使用者登入系統。

要求：
- 視覺樣式與桌面版登入頁一致。
- 版面需針對手機螢幕重排，不可直接縮小桌面版。
- 登入後導向手機版生產工單清單。

### 7.2 工單清單頁

目的：
- 提供現場共用工單池，讓人員快速找到要處理的工單。

主要內容：
- 搜尋：工單號、訂單號、客戶、品名 / 規格
- 篩選：狀態、日期、機台、是否急件
- 預設篩選建議採：
  - 狀態預設為 `待開工`、`生產中`、`暫停中`
  - 日期不預設
  - 機台不預設
- 清單顯示欄位：
  - 工單號
  - 客戶 / 品名 / 規格摘要
  - 指定機台
  - 狀態
  - 訂單號
  - 開始 / 完成時間
  - 現場照片數量
- 清單動作：
  - 開啟工單
  - 開工
  - 部分完工
  - 完工回報
  - 異常回報

### 7.3 工單執行頁

目的：
- 讓現場人員查看工單資訊並填寫本次生產回報。

顯示資訊：
- 工單號
- 訂單號
- 客戶簡稱
- 品名 / 規格
- 單重
- 預計淨重
- 預計支數
- 指定機台
- 工單狀態
- 若為拆分工單，顯示主工單號與機台份次資訊

可操作動作：
- 開工
- 暫停
- 恢復
- 部分完工回報
- 完工回報
- 異常回報
- 照片上傳

### 7.4 異常回報區塊 / 頁面

目的：
- 讓現場快速回報生產異常，不只寫在備註中。

建議欄位：
- 異常類型
- 發生時間
- 異常說明
- 照片
- 是否影響生產

### 7.5 照片上傳區塊

目的：
- 讓現場人員直接拍照回傳工單相關照片。

要求：
- 支援直接開啟手機相機
- 支援多張上傳
- 可預覽與刪除未送出照片
- 可選照片類型：
  - 完工照片
  - 生產中照片
  - 異常照片
  - 載具 / 成品照片
- 顯示已上傳縮圖、上傳時間與上傳者
- 上傳 modal 需在手機小螢幕維持可操作：
  - 內容區可捲動
  - 送出 / 取消按鈕需保持可見
  - 避免因預覽圖或欄位過高造成無法點擊下一步

### 7.6 桌面版讀取整合

目的：
- 讓辦公室生管人員在桌面版開啟工單時，能同步看到手機端回傳的現場照片。

第一階段原則：
- 桌面版先做唯讀整合。
- 先集中顯示於工單明細，不分散到多個桌面模組。
- 保留既有 `work_order_images` 顯示區塊，另外新增：
  - 完工圖片
  - 不良品圖片
  - 載具狀況圖片
- 每張圖片需顯示：
  - 縮圖
  - 上傳時間
  - 上傳者
  - 說明
  - 開啟原圖連結

## 8. 工單狀態流程草案

建議第一版手機端支援以下狀態：

- `待開工`
- `生產中`
- `暫停中`
- `部分完工`
- `已完成`

狀態動作建議：

- `待開工`：
  - 可開工
- `生產中`：
  - 可暫停
  - 可回報部分完工
  - 可回報完工
  - 可回報異常
- `暫停中`：
  - 可恢復生產
  - 可回報異常
- `部分完工`：
  - 可再次回報後續進度
  - 可回報最終完工
- `已完成`：
  - 原則不可再修改主要回報內容
  - 可補傳建議性照片或查看歷程

## 9. 欄位規劃

### 9.1 必要顯示欄位

- 工單號
- 訂單號
- 客戶名稱或簡稱
- 品名 / 規格
- 單重
- 預計淨重
- 預計支數
- 指定機台
- 狀態
- 排程日期

### 9.2 開工回報欄位

- 開工時間
- 操作人
- 備註

### 9.3 部分完工回報欄位

- 本次完成時間
- 本次完成淨重
- 備註

### 9.4 完工回報欄位

- 完工時間
- 備註
- 是否自動入庫
- 完工照片

### 9.5 異常回報欄位

- 異常類型
- 發生時間
- 異常說明
- 照片
- 是否影響生產

### 9.6 拆分工單 / 篩分場景擴充欄位

- 本機台完成淨重
- 本機台完成支數
- 各不良 / 服務項目數值
- 是否先入庫
- 現場補充說明

## 10. 照片上傳規則

- 完工照片為建議上傳，不強制。
- 異常照片建議保留為可上傳，但是否必填留待後續決策。
- 照片應與工單及操作紀錄建立關聯，不應只掛在單一文字欄位。
- 圖片應支援壓縮與格式限制，避免手機上傳過慢。

### 10.1 本輪資料表策略

- 為降低遠端測試影響，本輪不取代既有 `work_order_images`。
- 新增三張獨立圖片表，分別對應不同業務語意：
  - `work_order_completion_images`：完工圖片
  - `work_order_defect_images`：不良品圖片
  - `work_order_tool_condition_images`：載具狀況圖片
- 三張表均採一張工單對多張圖片設計。
- 第一版先不保留更細的事件關聯欄位，維持最小影響版本，只以 `work_order_id` 關聯。
- 欄位規劃原則可比照 `work_order_images`，至少需包含：
  - `id`
  - `work_order_id`
  - `file_name`
  - `file_path`
  - `file_size`
  - `mime_type`
  - `sort_order`
  - `description`
  - `uploaded_at`
  - `uploaded_by_employee_id`
  - `deleted_at`
- 更細緻事件欄位如：
  - `production_record_id`
  - `quality_issue_report_id`
  - `machine_run_id`
  - `inventory_item_id`
  本輪先不納入。

## 11. 權限規劃

### 11.1 第一版原則

- 目前相關使用者權限先一致，不區分操作員、班長、主管的手機版功能差異。
- 第一版不新增獨立權限鍵，先沿用既有登入與權限能力。

### 11.2 後續可預留權限鍵

- `mobile_work_orders.read`
- `mobile_work_orders.start`
- `mobile_work_orders.pause`
- `mobile_work_orders.report_progress`
- `mobile_work_orders.report_finish`
- `mobile_work_orders.upload_photo`
- `mobile_work_orders.report_issue`

## 12. 資料與追溯建議

- 手機版不應直接破壞既有 `work_orders` 主資料結構。
- 第一版確認補上 `工單執行紀錄`，作為現場操作追溯基礎。
- 異常追溯第一版先沿用既有 `quality_issue_reports`，不另建新的異常紀錄主表。
- 現場操作可追溯紀錄包含：
  - 工單執行紀錄
  - 工單照片附件
  - 品質異常回報
- 每次操作應記錄：
  - 工單
  - 操作類型
  - 操作人
  - 操作時間
  - 相關備註
  - 相關照片

### 12.1 第一版操作紀錄落地範圍

- 資料表：`work_order_operation_logs`
- 第一版先記錄以下動作：
  - 開工
  - 暫停
  - 恢復
  - 部分完工
  - 完工
  - 品質異常回報
  - 完工圖片上傳
  - 不良品圖片上傳
  - 載具狀況圖片上傳
- 工單明細 API 需回傳 `operation_logs`，供手機端查看現場追溯紀錄。

## 13. 與既有系統整合原則

- 手機版為既有 `work_orders` 的執行端，不是新的獨立工單體系。
- 工單主檔、訂單關聯與桌面版核心流程仍維持現有結構。
- 手機版新增的回報資料需與現有工單流程、列印、追溯與 DataSync 邏輯相容。
- 若後續修改入口或側邊欄，需同步調整 `index.html` 與 `index.php`。
- 手機版前端檔案優先集中於 `mobile/`，避免與桌面版入口檔混雜。

### 13.2 舊流程相容原則

- 遠端測試期間，舊桌面版流程與既有 API 不應被中斷。
- 新圖片表先平行新增，不做舊資料搬移。
- 舊的 `work_order_images` 保留原用途與讀取邏輯。
- 手機版新功能優先接新表；桌面版先補讀取能力，再評估是否追加管理能力。
- `api/work_orders/show.php` 後續需擴充回傳：
  - `completion_images`
  - `defect_images`
  - `tool_condition_images`
- 新圖片表異動後，需納入 DataSync，至少刷新 `work_orders` 工單明細相關畫面。

## 13.1 一鍵更新與部署路徑注意事項

- 一鍵更新包若包含手機版檔案，檔案清單必須保留 `mobile/` 相對路徑。
- 更新包內容不可把手機版檔案誤包到專案根目錄或其他桌面版目錄。
- 若手機版有額外靜態資源、子頁面或附件路徑，需確認遠端部署後可在 `https://mes.sort.com.tw/mobile` 底下正常存取。
- 若後續有 rewrite、登入導向或靜態資源 base path 設計，需優先檢查 `/mobile` 子路徑是否可正常運作。
