# MES 系統整體設計改善建議

> 建立日期：2026-02-09
> 基於資料庫實際結構與資料分析

---

## 一、業務流程缺口

### 1. 缺少「進料收貨」環節 — 最大缺口

目前流程直接從「訂單」跳到「工單」，但螺絲篩分業務的實際流程是：**客戶寄來螺絲 → 羽全收貨秤重確認 → 開工單生產**。

`order_items` 上雖有 `customer_provided_weight`（客戶申報重量）和 `confirmed_weight`（確認重量），但缺少一個正式的**進料收貨單 (Goods Received Note)** 來紀錄：

| 缺少的資訊 | 說明 |
|-----------|------|
| 收貨日期/時間 | 何時收到客戶材料 |
| 收貨人 | 誰負責收貨 |
| 秤重紀錄 | 毛重、淨重、載具扣重 |
| 外觀初檢 | 收到時螺絲是否有明顯異常 |
| 收貨照片 | 留存佐證 |
| 儲位安排 | 收貨後放在哪裡等待排產 |

**建議**：新增 `incoming_receipts`（進料收貨單）及 `incoming_receipt_items`（收貨明細），串在訂單品項與工單之間，完善物料追溯鏈。

### 2. 退貨→重工→再出貨的閉環未打通

目前 `return_order_items` 紀錄了退回數量，但退回後：
- 沒有自動建立庫存入庫交易
- 沒有機制產生「重工工單」
- `production_quality_records.rework_needed` 只是一個布林旗標，無法追蹤實際重工進度

**建議**：退貨完成後應自動觸發：
```
退貨入庫 → inventory_transactions (inbound/return)
         → 可選建立重工工單 (work_orders.type = 'rework')
         → 重工完成後再次入庫 → 再次出貨
```

### 3. 工單缺少「類型」區分

目前 `work_orders` 沒有 `type` 欄位，無法區分：
- 首次篩分工單 vs 重工工單 vs 返修工單
- 這影響生產統計的準確性（重工產出不應算入首次良率）

## 二、資料一致性問題

### 1. 狀態管理雙軌制 — 最嚴重的設計問題

從資料庫查詢結果可見，多個表同時存在 `status`（varchar）和 `status_lookup_id`（FK → lookup_values）：

| 資料表 | 狀態欄位 | 問題 |
|--------|---------|------|
| `orders` | `status` + `status_lookup_id` | 兩個欄位可能不同步 |
| `employees` | `status` + `status_lookup_id` | 同上 |
| `work_orders` | `status` + `status_lookup_id` | 同上 |
| `shipping_orders` | `status` + `status_lookup_id` + `return_status` + `has_return` | **4 個狀態欄位！** |
| `tools` | `status` + `status_lookup_id` | 同上 |

**實際資料已證實不一致**：`order_items` id=16 的 `total_shipped_quantity = 0` 且 `shipping_status = 'not_shipped'`，但實際出貨品項合計 `250,414` 件。

**建議**：
- 統一只用 `status_lookup_id` 作為單一真實來源（Single Source of Truth）
- `status` varchar 欄位若要保留，應透過觸發器或應用層嚴格同步
- `shipping_orders` 的 `has_return` 布林欄位是多餘的（`return_status != 'none'` 即可判斷）

### 2. 反正規化欄位缺少同步機制

| 冗餘欄位 | 位置 | 真實來源 | 風險 |
|---------|------|---------|------|
| `total_shipped_quantity` | `order_items` | `SUM(shipping_order_items.shipped_quantity)` | **已確認不一致** |
| `shipping_status` | `order_items` | 應由出貨品項推導 | 同上 |
| `quantity_shipped` | `inventory_items` | `SUM(shipping_order_items.shipped_quantity)` | 目前為 0 但實際有出貨 |
| `total_amount` | `orders` | `SUM(order_items.total_price)` | 手動運算可能落差 |

**建議**：建立資料庫觸發器（Trigger）或在 API 層統一封裝更新邏輯，確保以下操作時自動同步：
- 出貨單確認 → 更新 `order_items.total_shipped_quantity` + `inventory_items.quantity_shipped`
- 退貨確認 → 回沖上述欄位

## 三、資料庫結構問題

### 1. 軟刪除 (Soft Delete) 不一致

有 `deleted_at` 的表（15 個）和沒有的表對比：

| 有 deleted_at | 缺少 deleted_at（應該要有） |
|--------------|--------------------------|
| orders, work_orders, shipping_orders, return_orders | `order_items` — 無法軟刪除單一品項 |
| customers, suppliers, employees | `shipping_order_items` — 出貨品項無法軟刪除 |
| | `return_order_items` — 退貨品項無法軟刪除 |
| | `production_records` — 生產紀錄無法撤銷 |
| | `machine_maintenance_tasks` — 維修任務無法軟刪除 |

**建議**：所有業務資料表（非純紀錄性質的）都應有 `deleted_at`，至少包含上表右欄所列。

### 2. 編號序列 (`number_sequences`) 只有一筆

`number_sequences` 目前只有 `ORDER` 一筆，但系統實際使用 `ORDER`、`WO`、`SO`、`RO` 四種前綴。這表示工單、出貨單、退貨單的流水號可能是在 PHP 端硬編碼產生，未使用此表。

**建議**：將所有單號生成統一收斂到 `number_sequences` 表，包含：

```sql
INSERT INTO number_sequences (seq_key) VALUES 
('WO'), ('SO'), ('RO'), ('INV'), ('RCV');  -- 工單/出貨/退貨/庫存/收貨
```

### 3. `inventory_items` 過度反正規化

此表同時儲存了 `screening_item_id`、`work_order_id`、`order_item_id`、`order_id`、`customer_id` 五個 FK，但實際上透過 `work_order_id → order_item_id → order_id → customer_id` 一路 JOIN 即可取得。

**影響**：任何上游資料修改都需要同步更新此表，增加不一致風險。

**建議**：僅保留 `work_order_id` 作為主要關聯，其餘透過 JOIN 查詢。若查詢效能需要，可建立視圖 (View) 取代冗餘欄位。

## 四、品質追溯鏈斷裂

### 1. 品質異常報告 (`quality_issue_reports`) 缺乏強關聯

`issue_source_type` + `issue_source_id` 是一個多態關聯（Polymorphic Association），沒有外鍵約束。且 `issue_source_type` 包含 5 種來源（進料/生產/出貨/客戶退貨/內部稽核），但與相關表沒有 FK 保護。

**建議**：至少建立業務層面的驗證，確保 `issue_source_id` 指向的紀錄確實存在。

### 2. 出貨品質檢驗 (`shipping_quality_inspections`) 只關聯到出貨單級別

目前只有 `shipping_order_id`，無法區分是哪個出貨品項的異常。應增加 `shipping_order_item_id` 欄位。

## 五、流程自動化缺失

| 應自動化的環節 | 現況 | 建議 |
|--------------|------|------|
| 工單完成 → 自動入庫 | 需手動建立庫存項目 | 工單狀態改為「已完成」時自動建立 `inventory_items` + `inventory_transactions` |
| 出貨確認 → 更新訂單出貨狀態 | `order_items.shipping_status` 已證實未同步 | 出貨單確認時自動重新計算 |
| 退貨確認 → 庫存回沖 | 無此機制 | 退貨確認時自動建立入庫交易 |
| 訂單品項全數出貨 → 訂單狀態改完成 | 需手動 | 當所有品項 `shipping_status = 'fully_shipped'` 時自動更新 |
| 庫存出庫 → 更新在庫數量 | `inventory_items.quantity_on_hand` 未更新 | 出貨時自動扣減 |

---

## 六、優先改善排序

```
優先度 1（資料正確性）：修復狀態雙軌制 + 反正規化欄位同步
優先度 2（流程完整性）：新增進料收貨環節
優先度 3（自動化）：   建立工單完成→入庫→出貨的自動化鏈
優先度 4（追溯性）：   打通退貨→重工閉環
優先度 5（一致性）：   統一軟刪除 + 編號序列管理
```
