## 2. 發現問題清單

### 2.1 🔴 嚴重問題 (P1)

#### ISSUE-001: `shipping_orders/add_item.php` 事務處理問題

| 項目 | 內容 |
|------|------|
| **檔案** | `api/shipping_orders/add_item.php` |
| **問題描述** | 事務開始後，部分錯誤路徑沒有正確執行 rollback |
| **影響範圍** | 資料一致性 - 可能導致部分資料寫入而部分失敗 |
| **狀態** | ✅ 已修復 |
| **修復日期** | 2026-01-22 |
| **驗證日期** | 2026-01-22 |

**修復方式**：將所有驗證邏輯移至 `beginTransaction()` 之前執行，避免在事務中執行可能失敗的查詢後直接 return。包括：
- `getInventoryItem()` 查詢移至事務前
- `getShippingOrderDetails()` 查詢移至事務前
- 可用數量檢查移至事務前
- 重複品項檢查移至事務前

---

#### ISSUE-002: `inventory_items/delete.php` 缺少關聯檢查

| 項目 | 內容 |
|------|------|
| **檔案** | `api/inventory_items/helpers.php` |
| **問題描述** | 刪除庫存項目前未檢查是否有關聯的 `shipping_order_items` |
| **影響範圍** | 資料完整性 - 可能刪除已被出貨單引用的庫存項目 |
| **狀態** | ✅ 已修復 |
| **修復日期** | 2026-01-22 |
| **驗證日期** | 2026-01-22 |

**修復方式**：在 `canDeleteInventoryItem()` 函數中新增 `shipping_order_items` 關聯檢查，刪除前會檢查是否有出貨單項目引用此庫存品項。

---

### 2.2 🟡 中等問題 (P2)

#### ISSUE-003: `order_items/delete.php` 缺少 `work_orders` 關聯檢查

| 項目 | 內容 |
|------|------|
| **檔案** | `api/order_items/delete.php` |
| **問題描述** | 刪除訂單品項前未檢查是否有關聯的工單 |
| **影響範圍** | 業務邏輯 - CASCADE 刪除可能不是預期行為 |
| **狀態** | ✅ 已修復 |
| **修復日期** | 2026-01-22 |
| **驗證日期** | 2026-01-22 |

**修復方式**：在刪除前新增 `work_orders` 關聯檢查，若有工單引用此訂單品項則禁止刪除並回傳友善錯誤訊息。

---

#### ISSUE-004: `shipping_orders/update.php` 狀態回滾缺少庫存處理

| 項目 | 內容 |
|------|------|
| **檔案** | `api/shipping_orders/update.php` |
| **問題描述** | 出貨單狀態從 `confirmed` 改回 `draft` 時，未釋放已分配的庫存數量 |
| **影響範圍** | 庫存準確性 |
| **狀態** | ✅ 已修復 |
| **修復日期** | 2026-01-22 |
| **驗證日期** | 2026-01-22 |

**修復方式**：新增 `releaseInventoryForShippingOrder()` 函數，當出貨單狀態從「已確認」或「已出貨」變更回「草稿」時，自動將已分配的庫存數量加回原庫存品項。

---

#### ISSUE-005: 部分 API 缺少分頁參數上限驗證

| 項目 | 內容 |
|------|------|
| **檔案** | `api/audit_logs/index.php`, `api/inventory_transactions/index.php` |
| **問題描述** | `per_page` 參數沒有上限檢查，可能導致記憶體問題 |
| **影響範圍** | 系統穩定性 |
| **狀態** | ✅ 已確認 (原已修復) |
| **修復日期** | - |
| **驗證日期** | 2026-01-22 |

**驗證結果**：經檢查，兩個檔案皆已有 `min($perPage, 100)` 上限限制，此問題已修復。
