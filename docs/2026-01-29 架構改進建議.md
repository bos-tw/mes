# MES 系統架構改進建議

## 目標
- 降低模組間的耦合度
- 減少重複程式碼
- 方便未來維護與擴展

---

## 現況分析

### 目前架構

```
modules/
├── orders.html        (完整 HTML，約 200-400 行)
├── customers.html     (完整 HTML，約 200-400 行)
├── suppliers.html     (完整 HTML，約 200-400 行)
└── ... (30+ 個模組)

js/
├── orders.js          (業務邏輯)
├── customers.js       (業務邏輯)
└── ...
```

### 問題

| 問題 | 影響 | 嚴重度 |
|------|------|--------|
| HTML 結構重複 | 修改 header 需改 30+ 個檔案 | 🔴 高 |
| 容易不一致 | 如 sticky 問題 | 🟡 中 |
| 新增模組成本高 | 需複製大量樣板 | 🟡 中 |
| 難以批次更新 | 如按鈕樣式變更 | 🔴 高 |

---

## 建議方案：漸進式配置化

### 階段一：建立配置系統（已完成）

新增檔案：
- `core/module-config.js` - 模組配置註冊系統
- `core/module-renderer.js` - HTML 渲染引擎
- `core/configs/*.config.js` - 各模組配置

**優點**：
- 不影響現有程式碼
- 新模組可選擇使用新架構
- 可逐步遷移舊模組

### 階段二：遷移現有模組

將現有模組逐步改為配置化：

```javascript
// 舊方式：modules/customers.html (300行 HTML)

// 新方式：core/configs/customers.config.js (80行配置)
ModuleConfig.register('customers', {
    title: '客戶基本資料',
    subtitle: '維護客戶基本資料與聯絡資訊',
    actions: [
        { action: 'create', label: '新增', icon: 'fa-plus', style: 'primary' },
        // ...
    ],
    columns: [
        { key: 'name', label: '客戶名稱' },
        // ...
    ],
    // ...
});
```

### 階段三：整合到主系統

修改 `script.js` 的 `loadTabContent`：

```javascript
async function loadTabContent(tabContentElement, contentUrl, moduleId) {
    // 檢查是否有模組配置
    const config = ModuleConfig?.get(moduleId);
    
    if (config) {
        // 使用配置化渲染
        ModuleRenderer.renderTo(moduleId, tabContentElement);
    } else {
        // 舊方式：載入 HTML 檔案
        const response = await fetch(contentUrl);
        const html = await response.text();
        tabContentElement.innerHTML = html;
    }
    
    // 執行模組初始化
    runModuleInitializer(moduleId, tabContentElement);
}
```

## 架構比較

### 修改共用結構的比較

| 需求 | 舊架構 | 新架構 |
|------|--------|--------|
| 修改 header 按鈕樣式 | 改 30+ 個 HTML 檔 | 改 1 個模板函數 |
| 新增全域欄位 | 改 30+ 個 HTML 檔 | 改 1 個模板函數 |
| 調整 Modal 結構 | 改 30+ 個 HTML 檔 | 改 1 個模板函數 |
| 新增模組 | 複製 300 行 HTML | 寫 80 行配置 |

### 程式碼量比較

```
舊架構單一模組：
  modules/customers.html    ~350 行
  js/customers.js           ~500 行
  總計：~850 行

新架構單一模組：
  core/configs/customers.config.js  ~80 行
  js/customers.js                   ~500 行 (業務邏輯不變)
  總計：~580 行
  
  + 共用模板 (所有模組分攤)：~300 行
```

## 風險評估

| 風險 | 機率 | 影響 | 緩解措施 |
|------|------|------|----------|
| 新舊系統相容問題 | 中 | 中 | 保留舊載入方式作為 fallback |
| 配置語法錯誤 | 低 | 低 | 加入配置驗證機制 |
| 效能影響 | 低 | 低 | 渲染只在載入時執行一次 |

---

## 結論

建議採用**漸進式遷移**策略：

1. **短期**：新模組使用配置系統
2. **中期**：逐步遷移簡單模組
3. **長期**：全面配置化

這樣可以：
- 不影響現有功能
- 降低風險
- 逐步驗證新架構
- 保持系統穩定
