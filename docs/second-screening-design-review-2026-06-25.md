# 二次篩選設計重新檢討（2026-06-25）

> 本文件保留為設計討論紀錄。後續實作追蹤請以 `docs/second-screening-implementation-plan-2026-06-25.md` 與 `docs/second-screening-todo-2026-06-25.md` 為主。

## 結論

「二次篩選」應是一個可獨立追溯的品質流程節點，不應只被定義成退貨後的二次重篩，也不應只靠生產工單備註記錄。

系統需要支援同一個二次篩選模組下的兩種來源理由：

1. 異常後放寬標準重篩
   - 第一次篩選後不良過多。
   - 回報客戶後，客戶同意放寬標準。
   - 使用放寬後標準做第二次篩選，合格品再放行送回或出貨。

2. 客戶要求固定二次篩選
   - 客戶標準嚴格，每批訂單都必須經過第二次篩選。
   - 這不是異常，也不是退貨流程。
   - 第二次篩選是訂單/客戶規格要求的一部分。

兩者都叫「二次篩選」，差異在於「為什麼做第二次」與「來源如何產生」，不是兩套互不相干的功能。

## 設計原則

- 一批貨最多只做一次二次篩選，不設計第三次、第四次遞迴篩選。
- 生產工單頁必須能直接看出此批貨是否有二次篩選。
- 生產工單頁必須能直接看出二次篩選原因。
- 二次篩選必須有自己的篩選紀錄，不可只用工單備註、狀態或單一數字帶過。
- 二次篩選必須能追溯到原始訂單、原始工單、第一次篩選結果、第二次篩選標準與第二次篩選結果。
- 二次篩選模組可以獨立存在，但必須與生產工單互相導頁，不可變成品質模組裡孤立的歷史表。

## 建議統一資料語意

既有 `rescreen_batches` 可保留，但語意應調整為「二次篩選案件」而非只代表退貨後重篩。

建議命名與欄位語意：

- `second_screening_cases` 或沿用 `rescreen_batches`
  - 代表一批貨的第二次篩選案件。
  - 不限定來源一定是退貨單。

- `second_screening_reason`
  - `relaxed_after_high_defect`：第一次篩選不良過多，客戶同意放寬標準後重篩。
  - `customer_required_second_pass`：客戶/訂單要求固定二次篩選。

- `source_work_order_id`
  - 原始生產工單。

- `execution_work_order_id`
  - 第二次篩選的執行工單。
  - 若採同一張工單階段化，也可改為 `execution_stage_id`。

- `source_defect_history_record_id`
  - 僅在 `relaxed_after_high_defect` 需要。
  - 指向第一次篩選後造成二次篩選決策的不良紀錄。

- `customer_approval_reference`
  - 僅在放寬標準重篩時必填。
  - 可記錄客戶通知、核准備註或附件來源。

- `source_requirement_id`
  - 僅在固定二次篩選時使用。
  - 指向訂單品項、客戶規格或篩選要求設定。

## 篩選紀錄模型

二次篩選必須有明細紀錄，至少包含：

- 第二次篩選套用的服務項目。
- 原標準快照。
- 第二次篩選標準快照。
- 第二次篩選各服務的不良數量。
- 第二次篩選良品數量。
- 第二次篩選再次不良數量。
- 測量/篩選人員。
- 測量/篩選時間。
- 備註與附件。

固定二次篩選不一定有「放寬後標準」，但仍要保存第二次篩選使用的標準快照。

放寬標準重篩則必須清楚保存「原標準」與「放寬後標準」差異。

## 工單端應呈現的資訊

在生產工單列表與詳情/編輯畫面，至少要能看到：

- 是否有二次篩選：是/否。
- 二次篩選原因：
  - 不良過多，客戶同意放寬標準。
  - 客戶要求固定二次篩選。
- 二次篩選案件編號。
- 第二次篩選執行工單或階段。
- 第二次篩選結果摘要：
  - 良品數
  - 再次不良數
  - 報廢/其他處置數
- 可點進二次篩選詳情。

## 現有設計問題

目前已落地的第一輪設計有幾個明顯偏差：

- `rescreen_batches` 幾乎被綁定為退貨單後續流程。
- 建立流程預設從 `return_orders` 建立，無法自然承接「客戶要求每批固定二次篩選」。
- `rescreen_type = strict_rescreen / relaxed_rescreen` 只描述標準嚴格或放寬，沒有描述二次篩選的業務理由。
- 生產工單雖有 `source_rescreen_batch_id`，但目前比較像執行工單回掛案件，尚未成為原始工單上可一眼辨識的二次篩選狀態。
- 手動建立二次篩選執行工單的 API 能力存在，但畫面流程與案件來源語意尚未整理。

## 修正方向

1. 將模組顯示名稱從「二次重篩歷史紀錄」調整為「二次篩選紀錄」。
2. 將 `rescreen_type` 保留為標準型態，但新增或改用 `second_screening_reason` 表示業務理由。
3. 允許二次篩選案件從兩種入口建立：
   - 生產工單：客戶要求固定二次篩選。
   - 生產工單第一次篩選結果/不良紀錄：不良過多，客戶同意放寬標準。
4. 退貨單不再是二次篩選唯一入口；退貨來源只是一種可追溯來源。
5. 原始生產工單要能列出所有關聯二次篩選案件。
6. 二次篩選案件要保存第二次篩選紀錄與結果分流。
7. 出貨、庫存、不良品歷史要能追到二次篩選案件與原因。

## 實作優先順序

1. 先調整設計文件與 UI 用詞，避免繼續把二次篩選限縮成退貨重篩。
2. 補 `second_screening_reason` 類型與驗證。
3. 補從生產工單建立二次篩選案件的入口。
4. 在工單列表/詳情顯示二次篩選狀態、原因與結果摘要。
5. 讓二次篩選詳情頁呈現完整篩選紀錄。
6. 再整理既有退貨建立流程，讓它成為來源之一，而不是唯一來源。
