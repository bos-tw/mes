# 退貨單系統設計缺陷分析與改善方案

## 📌 執行摘要

經過徹底評估，**現有退貨單設計存在嚴重的結構性缺陷**，主要問題集中在**資料一致性、追溯完整性、業務邏輯完整性**三個面向。

### 🚨 嚴重性評級

| 缺陷 | 嚴重性 | 影響範圍 | 修復難度 |
|------|--------|----------|----------|
| [缺陷1] 追溯斷鏈 | 🔴 **致命** | 業務邏輯 | 高 |
| [缺陷2] 數據不一致風險 | 🔴 **致命** | 資料完整性 | 高 |
| [缺陷3] 分批出貨問題 | 🟠 **嚴重** | 業務流程 | 中 |
| [缺陷4] 數量追蹤缺失 | 🟠 **嚴重** | 庫存管理 | 中 |
| [缺陷5] 業務欄位不足 | 🟡 **中等** | 功能完整性 | 低 |

---

## 🔴 缺陷1：追溯斷鏈（致命）

### 問題描述

**退貨單主單** 和 **退貨單明細** 的追溯路徑不一致：

```
退貨單主單 (return_orders)
    ↓ original_shipping_order_id
出貨單 (shipping_orders)

退貨單明細 (return_order_items)
    ↓ order_item_id
訂單品項 (order_items)  ← 跳過了出貨單明細！
```

**正常應該是**：

```
退貨單明細 (return_order_items)
    ↓ shipping_order_item_id
出貨單明細 (shipping_order_items)
    ↓ order_item_id
訂單品項 (order_items)
```

### 實際影響

#### 情境A：分批出貨
```sql
-- 訂單品項：螺絲 1000 顆
INSERT INTO order_items (order_id, product_name, quantity)
VALUES (1, '螺絲M3', 1000);

-- 第一批出貨 600 顆（2026-01-10）
INSERT INTO shipping_order_items (shipping_order_id, order_item_id, shipped_quantity)
VALUES (1, 1, 600);

-- 第二批出貨 400 顆（2026-01-20）
INSERT INTO shipping_order_items (shipping_order_id, order_item_id, shipped_quantity)
VALUES (2, 1, 400);

-- 客戶退貨 100 顆
-- ❌ 問題：無法確定退的是第一批還是第二批！
INSERT INTO return_order_items (return_order_id, order_item_id, returned_quantity)
VALUES (1, 1, 100);  -- 只記錄 order_item_id，遺失了出貨批次資訊
```

**結果**：
- ✅ 知道退的是「螺絲M3」
- ❌ **不知道退的是哪一批出貨的貨**
- ❌ **無法追蹤每批出貨的退貨率**
- ❌ **可能導致庫存計算錯誤**

#### 情境B：不同出貨單包含相同訂單品項
```sql
-- 客戶回報：「2026-01-20 收到的貨有問題，要退貨」
-- 但系統只記錄 order_item_id，無法確定是哪張出貨單

SELECT * FROM return_order_items WHERE order_item_id = 1;
-- 結果：只能看到退貨數量，無法確定是哪次出貨
```

### 資料一致性風險

```sql
-- 退貨單主單說：這是退「出貨單#2」的貨
UPDATE return_orders SET original_shipping_order_id = 2 WHERE id = 1;

-- 但退貨單明細可能記錄的是「出貨單#1」的品項
-- 因為 return_order_items 只記錄 order_item_id，沒有檢查機制
INSERT INTO return_order_items (return_order_id, order_item_id, returned_quantity)
VALUES (1, 999, 50);  -- 這個 order_item_id 可能屬於出貨單#1！

-- ❌ 資料不一致！主單說退出貨單#2，明細卻是出貨單#1的品項
```

## 🔴 缺陷2：數據不一致風險（致命）

### 問題描述

**缺少約束條件確保退貨明細與退貨主單的一致性**。

### 表結構問題

```sql
CREATE TABLE return_order_items (
  return_order_id bigint NOT NULL,
  order_item_id bigint NOT NULL,  -- ❌ 只連到 order_items
  -- ❌ 缺少：shipping_order_item_id
  -- ❌ 缺少：CHECK 約束驗證 order_item 是否屬於該出貨單
  CONSTRAINT fk_return_order_items_order_item
    FOREIGN KEY (order_item_id) REFERENCES order_items(id)
);
```

### 可能發生的資料錯誤

#### 錯誤1：退貨品項與出貨單不符
```sql
-- 出貨單#1 包含：訂單品項 [10, 20, 30]
-- 出貨單#2 包含：訂單品項 [40, 50, 60]

-- 退貨單主單說：退出貨單#1
INSERT INTO return_orders (original_shipping_order_id) VALUES (1);

-- 但明細卻加入出貨單#2 的品項
INSERT INTO return_order_items (return_order_id, order_item_id)
VALUES (1, 40);  -- ❌ 品項40 不屬於出貨單#1！

-- ✅ 系統不會阻止這個操作！
```

#### 錯誤2：無法驗證退貨數量是否合理
```sql
-- 出貨數量：100 顆
INSERT INTO shipping_order_items (order_item_id, shipped_quantity)
VALUES (1, 100);

-- 退貨數量：150 顆（超過出貨數量！）
INSERT INTO return_order_items (order_item_id, returned_quantity)
VALUES (1, 150);  -- ❌ 退貨數量超過出貨數量，系統無法阻止！
```

#### 錯誤3：同一批貨被多次退貨
```sql
-- 出貨數量：100 顆
INSERT INTO shipping_order_items (id, order_item_id, shipped_quantity)
VALUES (1, 10, 100);

-- 第一次退貨：80 顆
INSERT INTO return_order_items (order_item_id, returned_quantity)
VALUES (10, 80);

-- 第二次退貨：50 顆（總共130顆，超過出貨數量！）
INSERT INTO return_order_items (order_item_id, returned_quantity)
VALUES (10, 50);  -- ❌ 累計退貨 130 顆 > 出貨 100 顆

-- 系統無法阻止，因為沒有追蹤 shipping_order_item_id
```

## 🟠 缺陷3：分批出貨問題（嚴重）

### 問題描述

**無法正確處理「同一訂單品項分多批出貨」的場景**。

### 實際案例

```sql
-- 訂單：客戶訂購螺絲 10,000 顆
INSERT INTO order_items (id, order_id, product_name, quantity, unit)
VALUES (1, 100, '螺絲M3x10', 10000, '顆');

-- 分三批出貨
INSERT INTO shipping_order_items (id, shipping_order_id, order_item_id, shipped_quantity, shipped_unit)
VALUES
  (1, 201, 1, 3000, '顆'),  -- 第一批 2026-01-05
  (2, 202, 1, 4000, '顆'),  -- 第二批 2026-01-12
  (3, 203, 1, 3000, '顆');  -- 第三批 2026-01-18

-- 客戶回報：2026-01-12 收到的貨（第二批）有瑕疵，要退 500 顆
-- ❌ 問題：系統無法記錄「退的是哪一批」
INSERT INTO return_order_items (return_order_id, order_item_id, returned_quantity)
VALUES (1, 1, 500);  -- 只知道退「訂單品項1」，不知道是哪批出貨

-- ❌ 後果：
-- 1. 品質追蹤：無法統計「第二批的不良率」
-- 2. 供應商責任：無法確定是哪次生產批次的問題
-- 3. 庫存管理：無法追蹤「哪批庫存有問題需要隔離」
```

### 查詢困難

```sql
-- ❌ 無法查詢：「出貨單#202 的退貨率」
SELECT
    soi.shipped_quantity,
    SUM(roi.returned_quantity) as total_returned  -- ❌ 錯誤！這會包含所有批次的退貨
FROM shipping_order_items soi
LEFT JOIN return_order_items roi ON soi.order_item_id = roi.order_item_id
WHERE soi.id = 2;  -- 第二批出貨

-- 結果：會把第一批、第三批的退貨也計入，數據錯誤！
```

## 🟠 缺陷4：數量追蹤缺失（嚴重）

### 問題描述

**缺少自動化機制追蹤「訂購→出貨→退貨」的數量關係**。

### 缺少的欄位與計算

#### order_items 缺少欄位
```sql
ALTER TABLE order_items ADD COLUMN (
  -- ❌ 缺少
  total_shipped_quantity DECIMAL(14,2) DEFAULT 0 COMMENT '累計已出貨數量',
  total_returned_quantity DECIMAL(14,2) DEFAULT 0 COMMENT '累計已退貨數量',
  remaining_quantity DECIMAL(14,2) COMMENT '剩餘未出貨數量 (計算欄位)',

  -- ❌ 缺少約束
  CONSTRAINT chk_shipped_not_exceed
    CHECK (total_shipped_quantity <= quantity),
  CONSTRAINT chk_returned_not_exceed
    CHECK (total_returned_quantity <= total_shipped_quantity)
);
```

#### shipping_order_items 缺少欄位
```sql
ALTER TABLE shipping_order_items ADD COLUMN (
  -- ❌ 缺少
  total_returned_quantity DECIMAL(14,2) DEFAULT 0 COMMENT '該批出貨的累計退貨數量',

  -- ❌ 缺少約束
  CONSTRAINT chk_return_not_exceed_shipped
    CHECK (total_returned_quantity <= shipped_quantity)
);
```

### 實際影響

```sql
-- 訂購 1000 顆
INSERT INTO order_items (quantity) VALUES (1000);

-- 出貨 1200 顆（超過訂購數量！）
INSERT INTO shipping_order_items (order_item_id, shipped_quantity)
VALUES (1, 1200);  -- ❌ 系統不會阻止！

-- 退貨 1500 顆（超過出貨數量！）
INSERT INTO return_order_items (order_item_id, returned_quantity)
VALUES (1, 1500);  -- ❌ 系統不會阻止！

-- ✅ 應該要有自動計算和約束檢查
```

## ✅ 改善方案

### 方案A：最小修改方案（向下相容）

**適用情境**：已有生產資料，無法大幅修改結構

#### 步驟1：新增欄位
```sql
-- 退貨單明細新增出貨單明細ID（允許NULL以相容舊資料）
ALTER TABLE return_order_items
ADD COLUMN shipping_order_item_id BIGINT NULL COMMENT '出貨單明細ID（新增）',
ADD KEY idx_shipping_order_item_id (shipping_order_item_id),
ADD CONSTRAINT fk_return_order_items_shipping_item
  FOREIGN KEY (shipping_order_item_id)
  REFERENCES shipping_order_items(id)
  ON DELETE RESTRICT;
```

#### 步驟2：新增觸發器驗證一致性
```sql
DELIMITER $$

CREATE TRIGGER trg_return_items_validate_consistency
BEFORE INSERT ON return_order_items
FOR EACH ROW
BEGIN
  DECLARE v_shipping_order_id BIGINT;
  DECLARE v_error_msg VARCHAR(255);

  -- 取得退貨單關聯的出貨單ID
  SELECT original_shipping_order_id INTO v_shipping_order_id
  FROM return_orders
  WHERE id = NEW.return_order_id;

  -- 如果有填 shipping_order_item_id，驗證是否屬於該出貨單
  IF NEW.shipping_order_item_id IS NOT NULL THEN
    IF NOT EXISTS (
      SELECT 1 FROM shipping_order_items
      WHERE id = NEW.shipping_order_item_id
      AND shipping_order_id = v_shipping_order_id
    ) THEN
      SET v_error_msg = CONCAT('退貨品項不屬於出貨單#', v_shipping_order_id);
      SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = v_error_msg;
    END IF;
  END IF;

  -- 驗證退貨數量不超過出貨數量
  IF NEW.shipping_order_item_id IS NOT NULL THEN
    IF NEW.returned_quantity > (
      SELECT shipped_quantity FROM shipping_order_items
      WHERE id = NEW.shipping_order_item_id
    ) THEN
      SIGNAL SQLSTATE '45000'
      SET MESSAGE_TEXT = '退貨數量不可超過出貨數量';
    END IF;
  END IF;
END$$

DELIMITER ;
```

#### 步驟3：新增視圖支援舊查詢
```sql
-- 建立視圖，讓舊程式碼仍可運作
CREATE OR REPLACE VIEW v_return_order_items_legacy AS
SELECT
  roi.*,
  soi.shipping_order_id,
  soi.shipped_quantity,
  o.order_number
FROM return_order_items roi
LEFT JOIN shipping_order_items soi ON roi.shipping_order_item_id = soi.id
LEFT JOIN order_items oi ON roi.order_item_id = oi.id
LEFT JOIN orders o ON oi.order_id = o.id;
```

### 方案B：結構重構方案（推薦）

**適用情境**：新系統或可接受資料遷移

#### 新表結構

```sql
-- 1. 退貨單明細改為連結出貨單明細
DROP TABLE IF EXISTS return_order_items;

CREATE TABLE return_order_items (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  return_order_id BIGINT NOT NULL COMMENT '退貨單ID',

  -- ✅ 改為連結出貨單明細（不是訂單品項）
  shipping_order_item_id BIGINT NOT NULL COMMENT '出貨單明細ID',

  -- 數量資訊
  returned_quantity DECIMAL(14,2) NOT NULL COMMENT '退貨數量',
  returned_unit VARCHAR(50) COMMENT '退貨單位',
  accepted_quantity DECIMAL(14,2) COMMENT '接受退貨數量',
  rejected_quantity DECIMAL(14,2) COMMENT '拒絕退貨數量',

  -- 退貨原因
  return_reason TEXT COMMENT '退貨原因',
  defect_type VARCHAR(100) COMMENT '瑕疵類型',
  defect_description TEXT COMMENT '瑕疵描述',

  -- 處理狀態
  is_reusable TINYINT(1) DEFAULT 1 COMMENT '是否可重新使用',
  inventory_action VARCHAR(50) COMMENT '庫存動作',
  restocked_quantity DECIMAL(14,2) COMMENT '已重新入庫數量',

  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

  -- 外鍵約束
  CONSTRAINT fk_return_items_return_order
    FOREIGN KEY (return_order_id)
    REFERENCES return_orders(id)
    ON DELETE CASCADE,

  CONSTRAINT fk_return_items_shipping_item
    FOREIGN KEY (shipping_order_item_id)
    REFERENCES shipping_order_items(id)
    ON DELETE RESTRICT,

  -- 數量約束
  CONSTRAINT chk_returned_qty_positive
    CHECK (returned_quantity > 0),
  CONSTRAINT chk_accepted_not_exceed
    CHECK (accepted_quantity <= returned_quantity),
  CONSTRAINT chk_rejected_not_exceed
    CHECK (rejected_quantity <= returned_quantity),
  CONSTRAINT chk_restocked_not_exceed
    CHECK (restocked_quantity <= accepted_quantity),

  INDEX idx_return_order_id (return_order_id),
  INDEX idx_shipping_order_item_id (shipping_order_item_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='退貨單明細';
```

#### 2. 新增數量追蹤欄位

```sql
-- shipping_order_items 新增退貨追蹤
ALTER TABLE shipping_order_items
ADD COLUMN total_returned_quantity DECIMAL(14,2) DEFAULT 0 COMMENT '累計退貨數量',
ADD CONSTRAINT chk_returned_not_exceed_shipped
  CHECK (total_returned_quantity <= shipped_quantity);

-- 觸發器自動更新
DELIMITER $$

CREATE TRIGGER trg_update_shipped_qty_on_return
AFTER INSERT ON return_order_items
FOR EACH ROW
BEGIN
  UPDATE shipping_order_items
  SET total_returned_quantity = total_returned_quantity + NEW.returned_quantity
  WHERE id = NEW.shipping_order_item_id;
END$$

CREATE TRIGGER trg_update_shipped_qty_on_return_delete
AFTER DELETE ON return_order_items
FOR EACH ROW
BEGIN
  UPDATE shipping_order_items
  SET total_returned_quantity = total_returned_quantity - OLD.returned_quantity
  WHERE id = OLD.shipping_order_item_id;
END$$

DELIMITER ;
```

#### 3. 建立追溯視圖

```sql
CREATE OR REPLACE VIEW v_return_traceability AS
SELECT
  -- 退貨單資訊
  ro.id AS return_order_id,
  ro.return_order_number,
  ro.return_date,
  ro.processing_status,

  -- 退貨明細
  roi.id AS return_item_id,
  roi.returned_quantity,
  roi.return_reason,

  -- 出貨單資訊（直接關聯）
  so.id AS shipping_order_id,
  so.shipping_order_number,
  so.shipping_date,

  -- 出貨明細
  soi.id AS shipping_order_item_id,
  soi.shipped_quantity,

  -- 訂單資訊（透過出貨明細追溯）
  o.id AS order_id,
  o.order_number,
  o.order_date,

  -- 訂單品項
  oi.id AS order_item_id,
  oi.product_name,
  oi.quantity AS ordered_quantity,

  -- 客戶資訊
  c.id AS customer_id,
  c.customer_name

FROM return_order_items roi
INNER JOIN return_orders ro ON roi.return_order_id = ro.id
INNER JOIN shipping_order_items soi ON roi.shipping_order_item_id = soi.id
INNER JOIN shipping_orders so ON soi.shipping_order_id = so.id
INNER JOIN order_items oi ON soi.order_item_id = oi.id
INNER JOIN orders o ON oi.order_id = o.id
LEFT JOIN customers c ON ro.customer_id = c.id

WHERE ro.deleted_at IS NULL;
```

### 方案C：混合方案（漸進式改善）

**階段1：立即執行（不影響現有功能）**
1. 新增 `shipping_order_item_id` 欄位（允許 NULL）
2. 新增業務欄位（檢驗狀態、退款金額等）
3. 建立追溯視圖
4. 更新 API 支援新欄位

**階段2：逐步遷移（3個月內）**
1. 修改前端介面，新增退貨時選擇「出貨明細」
2. 新資料強制填入 `shipping_order_item_id`
3. 舊資料透過腳本補全

**階段3：完全切換（6個月後）**
1. 將 `shipping_order_item_id` 改為 `NOT NULL`
2. 移除 `order_item_id` 欄位（改為透過 JOIN 取得）
3. 啟用所有約束和觸發器

## 📝 結論

現有退貨單設計存在**嚴重的結構性缺陷**，最關鍵的問題是：

1. **追溯斷鏈**：明細表直接連 order_items，跳過 shipping_order_items
2. **資料不一致**：缺少約束確保明細與主單的一致性
3. **分批出貨無法追蹤**：同一品項多次出貨時，無法確定退哪一批

**強烈建議採用方案C（混合方案）**，在不影響現有系統的前提下，分階段完成改善，6個月內達到最佳狀態。

如不處理這些問題，未來可能面臨：
- 📉 庫存數據不準確
- 📉 退貨率統計錯誤
- 📉 品質追蹤失效
- 📉 客戶糾紛無法釐清責任
- 📉 供應商求償困難
