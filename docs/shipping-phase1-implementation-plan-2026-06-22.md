# 出貨單第一階段實作規劃

更新日期：2026-06-22  
狀態：已完成，轉入第二階段規劃  
適用範圍：出貨單、出貨品項、出貨列印、出貨品質檢驗預留、不良品回送摘要、客戶載具摘要

## 1. 背景與目標

目前系統主流程已進入測試上線階段，`訂單 → 工單 → 庫存 → 出貨 → 退貨/沖銷` 中僅剩出貨單環節仍缺少實際可運作的第一版流程。

現場需求已明確：

- 出貨單必須可實際建立、列印與簽收。
- 紙本維持目前既有 A5 橫式版型，不另起新樣式。
- 產品出貨仍以庫存為主來源，不可脫離庫存帳。
- 不良品在第一階段只需記錄摘要，不在出貨單紙本列出完整明細。
- 客戶載具必須在出貨單紙本上留下本次歸還摘要，供現場簽收與後續追蹤。
- 出貨品質檢驗需保留資料結構與入口，但第一階段不強制綁死流程。

第一階段目標不是一次完成最終版總帳，而是先交付可運作的出貨單流程，讓使用者可開始實際操作，再依實務回饋逐步補強。

## 2. 第一階段 DoD

本輪最低可交付結果：

- 使用者可以建立一般產品出貨單。
- 出貨單可同時記錄：
  - 一般產品出貨品項
  - 不良品回送摘要
  - 客戶載具歸還摘要
- 列印頁維持現有 A5 橫式版型語言，只在既有結構上擴充欄位與區塊。
- 出貨單紙本可清楚顯示：
  - 產品出貨明細
  - 不良品摘要
  - 載具摘要
  - 合計與簽收欄
- 現有一般產品配貨與出貨庫存邏輯不被破壞。
- 出貨品質檢驗模組保留既有結構，第一階段不列為必填守門。

## 3. 第一階段範圍

### 3.1 會做

- 出貨單主檔新增出貨性質欄位。
- 出貨單新增不良品摘要資料結構。
- 出貨單新增客戶載具摘要資料結構。
- 出貨單畫面、詳情、列印同步支援上述欄位。
- 基本流程守門：
  - 一般產品必須來自庫存
  - 不良品摘要與載具摘要可手動輸入
  - 有不良品摘要時，出貨性質不可視為純一般出貨

### 3.2 第一階段不做滿

- 不良品完整總帳與隔離品庫存模型
- 退貨後二次重篩的完整自動串接
- 客戶載具往來總帳
- 出貨品質檢驗強制守門
- 依來源自動拆出多種紙本樣式

## 4. 資料模型建議

### 4.1 `shipping_orders`

建議新增：

- `shipment_purpose`
  - `normal`
  - `defect_return`
  - `tool_return`
  - `mixed`

用途：

- 標示本次出貨的主要性質。
- 作為前端顯示、列印抬頭與基本守門依據。

### 4.2 `shipping_order_items`

維持現有角色：

- 專門記錄一般產品出貨品項。
- 產品出貨仍由 `inventory_item_id` 回推：
  - 訂單
  - 訂單細項
  - 工單
  - 部分入庫
  - 原批號

第一階段不將不良品或載具硬塞進本表。

### 4.3 新增 `shipping_order_defect_summaries`

第一階段只做摘要，不做逐筆不良項目明細。

建議欄位：

- `id`
- `shipping_order_id`
- `source_shipping_order_id` nullable
- `source_work_order_id` nullable
- `source_inventory_item_id` nullable
- `defect_quantity`
- `weight_per_unit_g`
- `total_weight_kg`
- `notes`
- `created_at`
- `updated_at`

設計原則：

- 允許先記摘要，來源欄位可先選填或部分必填。
- 細節由獨立檢驗報告承接，不在出貨單紙本展開。

### 4.4 新增 `shipping_order_tool_summaries`

第一階段做本次出貨隨貨歸還的載具摘要。

建議欄位：

- `id`
- `shipping_order_id`
- `tool_id` nullable
- `tool_name`
- `tool_type`
- `unit_weight_kg`
- `quantity`
- `total_weight_kg`
- `notes`
- `created_at`
- `updated_at`

設計原則：

- 以本次歸還快照為主，不要求先做客戶載具總帳。
- 若後續要做客戶載具往來總帳，可由本表再擴充來源欄位。

## 5. 前端與操作流程

### 5.1 出貨單建立 / 編輯

出貨單 modal 建議新增三塊：

- 基本資訊
  - 出貨性質
  - 客戶
  - 關聯訂單
  - 出貨日期
  - 物流資訊
- 產品出貨明細
  - 延用現有從庫存挑選加入的模式
- 摘要區塊
  - 不良品摘要
  - 客戶載具摘要

### 5.2 不良品摘要輸入

第一階段只需要：

- 總數量
- 單重(g)
- 總重量(kg)
- 備註
- 可選來源欄位

不在第一階段要求輸入完整不良分類明細。

### 5.3 載具摘要輸入

第一階段以本次歸還摘要為主：

- 載具名稱
- 類型
- 數量
- 單重(kg)
- 總重(kg)
- 備註

若同一張單有多種載具，允許多列輸入。

## 6. A5 列印版型原則

### 6.1 必須遵守

- 沿用目前 `print/shipping_order_print.html` 的 A5 橫式版型。
- 延用目前頁首、標題、客戶資訊、明細表、備註、合計、簽收的整體視覺。
- 不另創 A4 版、不另創全新樣式。

### 6.2 第一階段紙本調整方向

保留主表為產品出貨明細，欄位優先順序如下：

- 項次
- 庫存編號
- 產品名稱
- 客戶批號
- 出貨數量
- 單位
- 良品重量(kg)
- 不良品重量(kg)
- 備註

在主表下方新增兩個摘要區塊：

#### 不良品摘要

- 不良品總數量
- 不良品單重(g)
- 不良品總重量(kg)
- 備註

#### 載具摘要

- 載具名稱 / 類型
- 數量
- 單重(kg)
- 總重(kg)
- 備註

### 6.2.1 重量顯示規範

- 第一階段出貨單畫面與 A5 列印中，所有重量欄位一律顯示至小數點後 2 位。
- 本規範適用於：
  - 良品重量(kg)
  - 不良品重量(kg)
  - 不良品單重(g)
  - 不良品總重量(kg)
  - 載具單重(kg)
  - 載具總重(kg)
  - 合計區所有重量欄位
- 數量欄位不套用本規範，仍依既有欄位語意顯示整數或既定精度。

### 6.3 紙本不做的內容

- 不良品細項分類
- 二次重篩說明
- 載具往來歷史
- 最低委託額度警示

上述資訊可保留於系統端或其他報表，不應擠入 A5 紙本。

## 7. 後端 API 方向

### 7.1 `api/shipping_orders/index.php`

- 建立出貨單時支援 `shipment_purpose`

### 7.2 `api/shipping_orders/show.php`

- 除既有產品品項外，回傳：
  - `defect_summary`
  - `tool_summaries[]`

### 7.3 `api/shipping_orders/update.php`

- 支援修改：
  - 出貨性質
  - 不良品摘要
  - 載具摘要

### 7.4 新增輔助 API

視現有架構決定是否拆為：

- `api/shipping_orders/save_defect_summary.php`
- `api/shipping_orders/save_tool_summaries.php`

若要降低複雜度，也可先整合到 `update.php` 同一 transaction 中處理。

## 8. 第一階段流程守門

### 8.1 必做守門

- 一般產品出貨必須來自 `inventory_item_id`
- 若有產品出貨，客戶必須與庫存客戶一致
- 不良品摘要數值不得為負
- 載具摘要數值不得為負
- `total_weight_kg` 應與數量與單重一致，若不一致以前端即時計算為準

### 8.2 第一階段先不阻擋

- 不良品摘要是否一定要有來源單號
- 客戶載具是否一定要對應既有 `tools`
- 是否必須先完成出貨品質檢驗才能出貨

上述先保留為第二階段強化項目。

## 9. 風險與折衷

### 9.1 已知折衷

- 第一階段不良品仍非完整總帳，只是可追溯摘要。
- 載具先做本次快照，不做欠還餘額計算。
- 退貨重篩先保留來源欄位，不在本輪完成閉環。

### 9.2 採用原因

- 目前系統已進入測試上線，必須優先交付可實際運作的出貨單。
- 若第一階段就做完整不良與載具總帳，風險高、範圍過大，會延誤整體交付。
- 先讓使用者跑出真實操作，再根據實際資料流補第二階段，較符合現況。
