## ✅ 正確的業務流程理解

### 螺絲篩分代工業務模式

**核心業務**：客戶送螺絲來，公司幫忙篩分檢驗，良品退回客戶。

```
客戶下單 → 收到貨 → 篩分檢驗 → 入庫 → 依交期出貨 → 客戶退貨（如有瑕疵）
```

### 完整資料流程

```
1. 訂單 (orders)
   ↓ 包含多個品項

2. 訂單品項 (order_items) - 客戶批號
   - 這是「客戶批號」，代表客戶送來的一批貨
   - 可能分多個載具運送
   ↓ 進行生產

3. 工單 (work_orders)
   - 根據訂單品項建立工單
   - 進行篩分檢驗作業
   ↓ 完工後

4. 庫存品項 (inventory_items)
   - 篩分完成的良品入庫
   - 一個訂單品項可能產生多個庫存品項
   - 關鍵欄位：order_id, work_order_id, screening_item_id
   ↓ 依交期出貨

5. 出貨單 (shipping_orders)
   ↓ 包含多個出貨品項

6. 出貨單明細 (shipping_order_items)
   - ✅ inventory_item_id：從哪個庫存品項出貨
   - ✅ order_item_id：對應到原始訂單品項（方便追蹤）
   - shipped_quantity：出貨數量
   ↓ 如果客戶發現問題

7. 退貨單 (return_orders)
   ↓ 包含多個退貨品項

8. 退貨單明細 (return_order_items)
   - ⚠️ order_item_id：只連到訂單品項
   - ❌ 缺少：shipping_order_item_id 或 inventory_item_id
```

## 🔍 關鍵發現：出貨單的實際設計

### shipping_order_items 表結構

```sql
CREATE TABLE shipping_order_items (
  id bigint NOT NULL,
  shipping_order_id bigint NOT NULL,      -- 出貨單ID
  order_item_id bigint NOT NULL,          -- 訂單品項ID（追溯用）
  inventory_item_id bigint DEFAULT NULL,  -- ★ 庫存品項ID（實際出貨來源）
  shipped_quantity decimal(14,2),
  shipped_unit varchar(50)
);
```

**實際資料範例**：
```
id: 17651152240785
shipping_order_id: 17651152243202
order_item_id: 16                 -- 對應到訂單品項（客戶批號）
inventory_item_id: 1              -- ★ 從庫存品項#1 出貨
shipped_quantity: 250414.00
```

### 這代表什麼？

**出貨單的建立邏輯**（從 add_item.php 看到的）：

1. 從「庫存品項」加入出貨單
2. 記錄 `inventory_item_id`（實際出貨的庫存）
3. 同時記錄 `order_item_id`（方便追溯到原始訂單）

**所以出貨單明細的設計是正確的！**
- ✅ 知道從哪個庫存品項出貨（`inventory_item_id`）
- ✅ 知道對應到哪個訂單品項（`order_item_id`）
- ✅ 記錄出貨數量

## 🚨 真正的問題：退貨單設計缺陷

### 問題 1：退貨單明細缺少關鍵連結

**現在的 return_order_items**：
```sql
CREATE TABLE return_order_items (
  id bigint NOT NULL,
  return_order_id bigint NOT NULL,
  order_item_id bigint NOT NULL,     -- ❌ 只連到訂單品項
  returned_quantity decimal(14,2),
  returned_unit varchar(50)
);
```

**缺少了什麼？**
```sql
-- ❌ 缺少 shipping_order_item_id - 不知道退的是哪次出貨
-- ❌ 缺少 inventory_item_id - 不知道退的是哪個庫存品項
```

### 為什麼這是問題？

#### 情境A：同一訂單品項，多次入庫、多次出貨

```
訂單品項#16（客戶批號：A123）- 訂購 1,000,000 支螺絲
  ↓
工單#1 完工 → 入庫品項#1（300,000 支）→ 2026-01-05 出貨#1
工單#1 完工 → 入庫品項#2（400,000 支）→ 2026-01-10 出貨#2
工單#2 完工 → 入庫品項#3（300,000 支）→ 2026-01-15 出貨#3
```

**客戶回報**：「1月10日收到的貨有瑕疵，要退 50,000 支」

**現在的退貨單明細**：
```sql
INSERT INTO return_order_items (return_order_id, order_item_id, returned_quantity)
VALUES (1, 16, 50000);
```

**問題**：
- ❌ 不知道退的是哪個入庫品項（品項1/2/3？）
- ❌ 不知道退的是哪次出貨（出貨1/2/3？）
- ❌ 無法追溯到是哪個工單生產的
- ❌ 無法統計「工單#1 的不良率」

### 問題 2：無法正確追溯品質問題

**品質問題追溯需要知道**：
1. 退貨的是哪個「庫存品項」→ 追溯到「工單」→ 追溯到「機台」、「篩分結果」
2. 退貨的是哪次「出貨」→ 追溯到「出貨日期」、「運輸方式」
3. 退貨的原始「訂單品項」→ 追溯到「客戶批號」、「客戶圖號」

**現在只能追溯到第3點，第1、2點完全無法追溯！**

### 問題 3：庫存管理混亂

**現在的設計**：
- 出貨時：從 `inventory_items` 扣除庫存
- 退貨時：應該要退回 `inventory_items`
- **但沒有記錄退回哪個庫存品項！**

**後果**：
1. 退貨後無法正確更新庫存
2. 無法統計「每個庫存品項的退貨率」
3. 無法隔離「有問題的批次庫存」

## ✅ 正確的改善方案

### 方案：修正 return_order_items 結構

```sql
-- 停用現有表（如果有資料，需先備份）
ALTER TABLE return_order_items RENAME TO return_order_items_old;

-- 建立新表結構
CREATE TABLE return_order_items (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  return_order_id BIGINT NOT NULL COMMENT '退貨單ID',

  -- ✅ 新增：連結到出貨單明細（最重要！）
  shipping_order_item_id BIGINT NOT NULL COMMENT '出貨單明細ID',

  -- 數量資訊
  returned_quantity DECIMAL(14,2) NOT NULL COMMENT '退貨數量',
  returned_unit VARCHAR(50) COMMENT '退貨單位',

  -- 退貨原因與處理
  return_reason TEXT COMMENT '退貨原因',
  defect_type VARCHAR(100) COMMENT '瑕疵類型',
  is_reusable TINYINT(1) DEFAULT 1 COMMENT '是否可重新使用',
  accepted_quantity DECIMAL(14,2) COMMENT '接受退貨數量',
  rejected_quantity DECIMAL(14,2) COMMENT '拒絕退貨數量',

  -- 庫存處理
  inventory_action VARCHAR(50) COMMENT '庫存動作（重新入庫/報廢/維修）',
  restocked_inventory_item_id BIGINT COMMENT '重新入庫的庫存品項ID',
  restocked_quantity DECIMAL(14,2) COMMENT '已重新入庫數量',

  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

  -- 外鍵約束
  CONSTRAINT fk_return_items_return_order
    FOREIGN KEY (return_order_id)
    REFERENCES return_orders(id)
    ON DELETE CASCADE,

  -- ✅ 關鍵外鍵：連到出貨單明細
  CONSTRAINT fk_return_items_shipping_item
    FOREIGN KEY (shipping_order_item_id)
    REFERENCES shipping_order_items(id)
    ON DELETE RESTRICT,

  CONSTRAINT fk_return_items_restocked_inventory
    FOREIGN KEY (restocked_inventory_item_id)
    REFERENCES inventory_items(id)
    ON DELETE SET NULL,

  -- 數量約束
  CONSTRAINT chk_returned_qty_positive
    CHECK (returned_quantity > 0),
  CONSTRAINT chk_accepted_not_exceed
    CHECK (accepted_quantity IS NULL OR accepted_quantity <= returned_quantity),
  CONSTRAINT chk_rejected_not_exceed
    CHECK (rejected_quantity IS NULL OR rejected_quantity <= returned_quantity),

  INDEX idx_return_order_id (return_order_id),
  INDEX idx_shipping_order_item_id (shipping_order_item_id),
  INDEX idx_restocked_inventory (restocked_inventory_item_id)
) ENGINE=InnoDB COMMENT='退貨單明細';
```

### 為什麼這樣設計？

#### ✅ 完整的追溯鏈

```
退貨單明細 (return_order_items)
  ↓ shipping_order_item_id
出貨單明細 (shipping_order_items)
  ↓ inventory_item_id
庫存品項 (inventory_items)
  ↓ work_order_id
工單 (work_orders)
  ↓ order_item_id
訂單品項 (order_items)
  ↓ order_id
訂單 (orders)
```

**可以追溯到**：
- ✅ 哪次出貨（shipping_order_item → shipping_order）
- ✅ 哪個庫存品項（shipping_order_item → inventory_item）
- ✅ 哪張工單生產的（inventory_item → work_order）
- ✅ 使用哪台機台（work_order → production_records → machine）
- ✅ 原始訂單品項（inventory_item → order_item）
- ✅ 客戶批號、圖號（order_item）

#### ✅ 正確的庫存管理

```sql
-- 退貨處理流程
-- 1. 建立退貨單明細，連結到出貨單明細
INSERT INTO return_order_items (
  return_order_id,
  shipping_order_item_id,  -- ✅ 記錄從哪次出貨退回
  returned_quantity
) VALUES (1, 17651152240785, 50000);

-- 2. 檢驗後，決定是否重新入庫
-- 如果可重新使用，建立新的庫存品項
INSERT INTO inventory_items (
  customer_id,
  order_id,
  work_order_id,  -- ✅ 可以追溯到原始工單
  source_type,    -- 'return'
  source_id,      -- return_order_item_id
  quantity_on_hand,
  quality_status  -- 'quarantine' 待檢驗
) VALUES (...);

-- 3. 更新退貨單明細，記錄重新入庫的庫存ID
UPDATE return_order_items
SET restocked_inventory_item_id = {新庫存ID},
    restocked_quantity = 50000,
    inventory_action = '重新入庫'
WHERE id = 1;
```

### 與現有 shipping_order_items 的一致性

**對比**：
```sql
-- shipping_order_items (出貨單明細)
shipping_order_id       -- 主單
inventory_item_id       -- ✅ 從哪個庫存品項出貨
order_item_id           -- 對應訂單品項（追溯用）
shipped_quantity        -- 出貨數量

-- return_order_items (退貨單明細) - 新設計
return_order_id              -- 主單
shipping_order_item_id       -- ✅ 從哪次出貨退回（對應 shipping_order_items）
returned_quantity            -- 退貨數量
restocked_inventory_item_id  -- ✅ 重新入庫到哪個庫存品項
```

**邏輯一致性**：
- 出貨：從 `inventory_items` 出去 → 記錄在 `shipping_order_items.inventory_item_id`
- 退貨：退回某次出貨 → 記錄在 `return_order_items.shipping_order_item_id`
- 重新入庫：退回的貨入庫 → 記錄在 `return_order_items.restocked_inventory_item_id`

## ✅ 結論

### 主要問題

**退貨單明細只連 order_item_id，缺少 shipping_order_item_id**，導致：
1. 🔴 **無法追溯到具體哪次出貨**
2. 🔴 **無法追溯到哪個庫存品項**
3. 🔴 **無法追溯到哪張工單、哪台機台**
4. 🔴 **無法正確進行品質分析**
5. 🔴 **無法正確管理退貨庫存**

### 解決方案

**新增 `shipping_order_item_id` 欄位，連結到出貨單明細**，實現：
- ✅ 完整追溯：退貨→出貨→庫存→工單→訂單
- ✅ 品質分析：統計每批次、每工單、每機台的退貨率
- ✅ 庫存管理：正確處理退貨入庫
- ✅ 設計一致性：與 shipping_order_items 邏輯對應
